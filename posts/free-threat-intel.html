Threat Intel for $0: Building a Passive Dashboard That Actually Blocks

I got tired of tabbing between CISA's KEV catalog, abuse.ch, and a dozen bookmarks every time I wanted to know what was actively being exploited. So I built a tool to pull it all into one place. No subscription. No API key. No enterprise budget.

The result is live at passive-intel.vercel.app. The source is at github.com/noisyloop/passive-intel.

This post covers how it works, what it surfaces, and how to actually use that intel to block threats on your own infrastructure.

What It Does

Three panels, all read-only, all passive:

  • CISA KEV — the Known Exploited Vulnerabilities catalog, newest first. Filterable by vendor, product, or CVE ID. Click any entry to expand the description and pull a live CVSS base score from NVD. Ransomware-linked entries are flagged.
  • URLhaus (online feed) — live malware URLs actively serving payloads right now, not historical. Filterable by host, threat type, or tag.
  • Feodo Tracker — active botnet command-and-control IPs with malware family and country. The same feed powering a lot of commercial threat intel platforms, available for free.

Nothing here scans anything. No probes, no active recon. Every data point is cached public intelligence pulled server-side. The browser only ever talks to the app itself.

The Free APIs Doing the Heavy Lifting

Credit where it's due — none of this works without these:

  • CISA KEV — confirmed in-the-wild exploitation, updated frequently. Free JSON feed, no auth.
  • URLhaus by abuse.ch — community-sourced malware URL feed, online-only subset. Free CSV, no auth.
  • Feodo Tracker by abuse.ch — active botnet C2 IPs covering Emotet, QakBot, AsyncRAT, and others. Free JSON, no auth.
  • NVD — CVSS base scores from NIST. Free REST API, no auth required for reasonable usage.

A note on secrets

There are no API keys in this codebase — not because I forgot them, but because I deliberately chose feeds that don't require them. This matters more than it sounds.

A common mistake when replicating this pattern is reaching for a paid or credentialed API and then hardcoding the key client-side or committing it to a public repo. Don't. If you extend this with a feed that requires authentication, that key belongs in an environment variable on the server side — never in frontend code, never in version control. The reason this stack is clean is that the feeds are free and open by design. The moment you add a credentialed API, the architecture has to account for it.

Private API keys hardcoded in client-side JavaScript or committed to a public repo are not a configuration choice — they're a credential exposure. Free feeds sidestep this entirely. If you're adding paid APIs, keep secrets server-side.

The Security Posture

The feeds contain attacker-influenced data — live malware URLs, C2 IPs, threat actor tags. The client code is deliberately paranoid about how it renders them.

Every dynamic value is written via textContent, never via innerHTML or any markup-building path. Malware URLs are displayed as inert plain text — no <a href> anywhere. A poisoned feed entry containing a script tag renders as visible text, not code. The CSP reinforces this as a second independent layer: default-src 'none', script-src 'self', connect-src 'self'.

The API functions reject scanner traffic before any handler logic runs. Missing or known-bad User-Agent strings get a 403 immediately. Rate limiting is per-IP per-endpoint. Unknown paths return a clean 404 from a dedicated handler — nothing useful leaks about routing.

Actually Blocking the Threats

The intel is only useful if you do something with it. The app exposes /api/blocklist.txt — one Feodo C2 IP per line, pulled from the live feed and cached at the edge:

https://passive-intel.vercel.app/api/blocklist.txt

Point any of the scripts below at it, or hit the Feodo recommended list directly. Either works.

nginx

#!/bin/bash
# /etc/cron.hourly/update-feodo-blocklist
curl -sf https://feodotracker.abuse.ch/downloads/ipblocklist_recommended.txt \
  | grep -v '^#' \
  | sed 's|^|deny |; s|$|;|' \
  > /etc/nginx/conf.d/feodo-blocklist.conf
nginx -t && nginx -s reload

Include it in your http or server block:

include /etc/nginx/conf.d/feodo-blocklist.conf;

Run hourly. The list rotates — stale blocks are harmless, missing new ones is not.

ufw

#!/bin/bash
curl -sf https://feodotracker.abuse.ch/downloads/ipblocklist_recommended.txt \
  | grep -v '^#' \
  | while read ip; do
      ufw deny from "$ip" to any
    done

Same cron cadence. Works on any Linux host you control directly.

Cloudflare (free tier)

CF's free tier supports IP Access Rules via API. Set your credentials as environment variables — not hardcoded in the script:

#!/bin/bash
# CF_ZONE and CF_TOKEN set as env vars, not hardcoded
curl -sf https://feodotracker.abuse.ch/downloads/ipblocklist_recommended.txt \
  | grep -v '^#' \
  | while read ip; do
      curl -s -X POST \
        "https://api.cloudflare.com/client/v4/zones/$CF_ZONE/firewall/access_rules/rules" \
        -H "Authorization: Bearer $CF_TOKEN" \
        -H "Content-Type: application/json" \
        --data "{\"mode\":\"block\",\"configuration\":{\"target\":\"ip\",\"value\":\"$ip\"},\"notes\":\"feodo-c2\"}"
    done

Free tier caps at 200 rules per zone. Filter to the recommended subset or prioritize by malware family if you hit the limit.

What This Isn't

This isn't a replacement for a SIEM, an EDR, or a full threat intel platform. It doesn't automatically push rules to your firewall — though /api/blocklist.txt gives you a stable feed URL a cron job can act on, which is close enough for most self-hosted setups.

What it is: a fast, free, passive view of what's actively being exploited and what infrastructure is confirmed malicious right now. Good for a morning check, a quick lookup mid-incident, or as a feed source for the blocking scripts above.

The person standing up their first server doesn't need a $50k threat intel subscription. They need to know these feeds exist and that there's a one-liner to act on them.

Total monthly cost: $0. Sources: CISA, abuse.ch, NVD. No database. No auth. No environment variables required.
